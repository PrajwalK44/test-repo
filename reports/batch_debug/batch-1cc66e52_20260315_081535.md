# Incident Report: batch-1cc66e52

## 1. Issue
**SQL Injection Vulnerability in Product Search Endpoint**
The `search_products()` function in `products.py` was using **string formatting** (`f-string`) to directly interpolate user-provided input (`query`) into a raw SQL query. This created a **critical SQL injection vulnerability**, allowing attackers to inject malicious SQL via the `q` query parameter.

**Vulnerable Code:**
```python
sql = f"SELECT * FROM products WHERE name LIKE '%{query}%' OR description LIKE '%{query}%'"
```
- **Risk**: Arbitrary SQL execution (e.g., `DROP TABLE`, data exfiltration, or authentication bypass).
- **Trigger**: Any request to `/products/search?q=<malicious_payload>` could exploit this.

---

## 2. Solution
**Parameterized Queries with SQLAlchemy**
The fix replaced the unsafe string interpolation with **SQLAlchemy’s parameterized query binding** (`:query` placeholder + `{"query": value}`). This ensures:
1. User input is treated as **data**, not executable SQL.
2. The database driver handles proper escaping/quoting.
3. Compliance with OWASP SQL injection prevention guidelines.

**Fixed Code:**
```python
sql = "SELECT * FROM products WHERE name LIKE :query OR description LIKE :query"
results = db.session.execute(db.text(sql), {"query": f"%{query}%"})
```

---

## 3. Applied Approach
1. **Identification**:
   - Static code analysis flagged the `f-string` in SQL context (bandit/Semgrep rule `G200`).
   - Manual review confirmed the vulnerability in `search_products()`.

2. **Remediation Steps**:
   - Replaced `f-string` with a **named parameter** (`:query`).
   - Passed the query value as a **bound parameter** in the `execute()` call.
   - Verified the `%` wildcards were safely included in the parameter value (not the SQL string).

3. **Validation**:
   - Tested with benign inputs (e.g., `q=test`).
   - Tested with SQL injection payloads (e.g., `q='; DROP TABLE products--`) to confirm mitigation.
