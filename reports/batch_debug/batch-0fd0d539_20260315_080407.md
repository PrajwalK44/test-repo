# Incident Report: Batch `0fd0d539` - Authentication Failure Due to Password Hash Encoding Mismatch

---

### **1. Issue**
The authentication system failed during password validation due to an **encoding mismatch** in `bcrypt.checkpw()`. The error occurred because:
- `bcrypt.checkpw()` expects **both the input password and stored hash to be `bytes`**, but the stored `user.password_hash` (retrieved from the database) was a **Unicode string** (`str`).
- This caused `bcrypt.checkpw()` to silently fail (return `False`) even for correct passwords, as it compared `bytes` (input) against `str` (stored hash).

**Symptoms**:
- Users could not log in despite correct credentials.
- No explicit error was logged (silent failure in `bcrypt.checkpw`).
- Incident triggered during batch debugging (`batch-0fd0d539`) for `incident_workdir_DEV-74_2b866919`.

**Root Cause**:
Missing `.encode("utf-8")` on the stored `user.password_hash` in `auth.py`.

---

### **2. Solution**
The fix ensured **consistent encoding** by converting the stored `user.password_hash` (Unicode `str`) to `bytes` before passing it to `bcrypt.checkpw()`. This aligns with `bcrypt`’s requirement that both arguments must be `bytes`.

**Key Change**:
```python
# Before: user.password_hash (str) passed directly
is_valid = bcrypt.checkpw(data["password"].encode("utf-8"), user.password_hash)

# After: user.password_hash encoded to bytes
is_valid = bcrypt.checkpw(data["password"].encode("utf-8"), user.password_hash.encode("utf-8"))
```

---

1. **Reproduction**:
   - Simulated login with correct credentials in the debug environment (`incident_workdir_DEV-74_2b866919`).

2. **Diagnosis**:
   - Inspected `user.password_hash` type (Unicode `str`) vs. expected `bytes` in `bcrypt` docs.

3. **Fix Implementation**:
