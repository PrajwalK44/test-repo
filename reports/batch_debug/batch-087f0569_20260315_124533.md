# Incident Report: Batch 087f0569

## 1. Issue
A **500 Internal Server Error** occurred on `POST /api/payments/checkout` in the staging environment. The root cause was traced to an **authentication subsystem failure** where `bcrypt.checkpw()` was incorrectly comparing a UTF-8 encoded password against a **non-encoded** stored password hash (`user.password_hash`). This mismatch caused authentication failures for valid credentials, propagating into payment workflows that required JWT validation.

**Key symptoms**:
- Payment checkout endpoint returned 500 errors despite valid requests.
- Authentication flow silently failed due to `bcrypt.checkpw()` receiving mismatched data types (bytes vs. non-encoded hash).
- Debug logs showed no explicit error, but the agent identified the issue via static analysis of `auth.py`.

---

## 2. Solution
The fix involved **encoding the stored password hash** (`user.password_hash`) to UTF-8 before passing it to `bcrypt.checkpw()`. This ensures both arguments to `checkpw()` are byte strings, as required by the bcrypt library.

**Why this works**:
- `bcrypt.checkpw()` expects both the input password and stored hash to be `bytes` (not `str`).
- The input password was already encoded (`data["password"].encode("utf-8")`), but the stored hash was not.
- Encoding the stored hash aligns the data types and resolves the silent comparison failure.
