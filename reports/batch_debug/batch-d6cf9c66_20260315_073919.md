# **Incident Report: Batch d6cf9c66**

## **1. Issue**
A **`TypeError`** occurred during user authentication in the `/login` endpoint due to **incompatible byte/string comparison** in `bcrypt.checkpw()`. The function expects both the input password and stored hash to be in **bytes format**, but the stored `user.password_hash` was being passed as a **UTF-8 string**, causing authentication failures.

**Error Context:**
- **File:** `python-service/app/routes/auth.py` (Line 55)
- **Function:** `login()`
- **Root Cause:** `bcrypt.checkpw()` received a **string** (`user.password_hash`) instead of **bytes**, leading to a silent failure where valid credentials were rejected.

**Impact:**
- Users could not log in despite correct credentials.
- No explicit error was logged (only implicit failure in authentication flow).

---

## **2. Solution**
The fix involved **encoding the stored password hash** (`user.password_hash`) to **UTF-8 bytes** before passing it to `bcrypt.checkpw()`, ensuring type consistency with the input password (already encoded).

**Key Change:**
```python
bcrypt.checkpw(
    data["password"].encode("utf-8"),
    user.password_hash.encode("utf-8")  # <-- Added `.encode("utf-8")`
)
```

---

## **3. Applied Approach**
1. **Reproduced the Issue:**
   - Simulated a login request with valid credentials.
   - Confirmed `bcrypt.checkpw()` returned `False` due to type mismatch.

2. **Root Cause Analysis:**
   - Inspected `bcrypt.checkpw()` documentation: **both arguments must be bytes**.
   - Verified `user.password_hash` was stored as a **string** in the database.

3. **Fix Implementation:**
   - Modified `auth.py` to encode `user.password_hash` to bytes before comparison.

4. **Validation:**
   - Re-tested login with the same credentials → **successful authentication**.
   - Verified no regression in other auth-related endpoints.
